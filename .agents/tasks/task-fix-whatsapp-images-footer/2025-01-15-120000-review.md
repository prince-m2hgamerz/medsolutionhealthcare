# Home page cleanup: images, duplicate WhatsApp button, and footer layout

This change removes a duplicate floating WhatsApp button from the home page (one already lives in AppShell), replaces third-party and AI-generated image URLs across five home-page sections with Unsplash equivalents, adds `images.unsplash.com` to the Next.js remote patterns config, and restructures the footer into a two-column (brand + 4-col link grid) layout with social media icons and improved mobile spacing.

Watch for: Two treatment cards share the same Unsplash photo URL (confirmed). Footer social links are hardcoded instead of pulling from `useSiteSettings()` which already exposes `facebook_url`, `instagram_url`, `youtube_url` (confirmed). The dead `WhatsAppFloat.tsx` file was left on disk (confirmed).

## High-level view

The WhatsApp fix removes the page-level import and render, relying on the AppShell's global `FloatingButtons`. The original `WhatsAppFloat.tsx` component file remains on disk as dead code with zero import references.

"Knee Replacement Surgery" and "Brain Tumor Surgery" cards ended up with an identical Unsplash image (`photo-1559757148-5c350d0d3c56`), so they'll appear visually duplicated when scrolling through treatments.

The footer restructure moves from a 5-column flat grid to a nested layout (`[1fr_3fr]` at `lg`, collapsing to a 2-column link grid on mobile). Social media icons are inline SVGs with accessible labels, but the URLs are hardcoded strings that bypass the dynamic `useSiteSettings()` hook the footer already invokes — admin-configured URLs won't propagate to these icons.

The `package-lock.json` diff removes `libc` constraint entries from optional native dependencies, unrelated to the feature work.

## Duplicate Unsplash image in TreatmentCostShowcase

The `treatmentCosts` array assigns `https://images.unsplash.com/photo-1559757148-5c350d0d3c56?w=600&q=80` to both "Knee Replacement Surgery" (index 1) and "Brain Tumor Surgery" (index 6). A user scrolling through treatment cards will see an identical photo for two unrelated procedures:

```tsx
// index 1
{ name: "Knee Replacement Surgery", image: "…photo-1559757148-5c350d0d3c56…" },
// index 6
{ name: "Brain Tumor Surgery",      image: "…photo-1559757148-5c350d0d3c56…" },
```

One of these should be swapped for a different image.

## Footer social links bypass settings

The footer component destructures `useSiteSettings()`:

```tsx
const { site_name, contact_phone, contact_email } = useSiteSettings();
```

The hook also returns `facebook_url`, `instagram_url`, and `youtube_url`, which are admin-configurable via the settings page. But the new social icon links are hardcoded:

```tsx
<a href="https://facebook.com/medsolutionhealthcare" …>
<a href="https://instagram.com/medsolutionhealthcare" …>
<a href="https://youtube.com/@medsolutionhealthcare" …>
```

If an admin updates the URLs via the settings panel, the footer icons won't reflect the change. Fix: destructure `facebook_url`, `instagram_url`, `youtube_url` from the existing hook call and use them in the `href` attributes.

## Dead WhatsAppFloat component

No file imports `components/shared/WhatsAppFloat.tsx` after this change (grep confirms zero references). The file still exists on disk and should be deleted to avoid confusion.

## Footer touch target size

The social icons use `w-9 h-9` (36px). Horizontal effective tap area is fine (36px + 12px gap = 48px), but vertical touch area is exactly 36px — below the 44px minimum recommended by WCAG 2.5.8. Borderline, not a blocker, but worth noting for an accessibility pass.
